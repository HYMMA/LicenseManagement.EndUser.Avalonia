# Webapp fixes that address `audit-2026-05-15.md`

**Date:** 2026-05-15
**Webapp branch:** `audit/fix` (worktree at `C:\Users\Ashra\source\repos\audit-fix`)
**Scope:** Server-side changes made in the `LicenseManagement` webapp in response to findings in the Avalonia wrapper audit.

The Avalonia wrapper audit calls out a lot of UI-local concerns (async-void event handlers, RelayCommand `CanExecuteChanged` never raised, cross-thread `StyledProperty` assignment, emoji icons, etc.). Almost all of it is local. The places where it indirectly points back at the webapp are below.

---

## Findings indirectly addressed in the webapp

| Audit finding | Severity | Webapp change |
|---|---|---|
| Lens 4 — "Branch on the documented exceptions and provide differentiated UX: `ApiException` with 429 → 'rate limited, retrying in N seconds'." | High | The webapp now emits **`Retry-After`** on every 429 from the auth rate limiter (`Utilities/AppConfig/SecurityMiddleware.cs:24-44`). Avalonia VMs can read this through the SDK's `ApiException` and show "Retrying in N seconds". |
| Lens 4 — "Branch on `ApiException` with 5xx → 'service temporarily unavailable, try later'." | High | The webapp now returns **RFC 7807 problem+json** on every non-2xx (`Program.cs:30-35`, `ServiceProviders.cs:50-65`). 5xx responses carry a stable `detail` string the SDK can surface verbatim. |
| Lens 4 — "There is no operation/correlation ID propagated. If a customer reports 'I tried to register at 14:32 UTC', support has no way to find the corresponding API request." | Medium | Every API response now carries **`X-Correlation-Id`** (echoes inbound if the wrapper provides one). `Utilities/AppConfig/SecurityMiddleware.cs:104-118`. The wrapper should surface it in `ErrorViewModel`. |
| Lens 4 — "Double-submit on Register: `CanExecute` does not refresh while in-flight." Server-side mitigation: idempotency. | Medium | New **`Idempotency-Key` middleware** (`Utilities/AppConfig/IdempotencyMiddleware.cs`). The SDK should send `Idempotency-Key = ReceiptCode + Computer.MacAddress` on `POST /api/license`; the webapp caches the response for 24 h so a duplicate submit never burns a seat. |
| Lens 4 — "Pass via `PublisherPreferences` metadata (the SDK supports `BeforeLicensePost` metadata events)" — implies the server accepts arbitrary metadata per license. | Medium | Already supported: `POST /api/license` accepts a `Metadata: Dictionary<string,string>` field today (see `LicenseManagement/Models/LicensePostApiModel.cs` and the `Metadata` column on the License table). No change required on the webapp side — confirming for the wrapper that it can rely on this without bumping a server version. |
| Lens 4 — "No observability — there is no `ILogger` injection, no `EventSource`, no telemetry." | Medium | Server-side, Serilog already captures every request. The new correlation id closes the loop so the wrapper's logs can join with ours. |
| Lens 3 (Security) — "Exception messages are shown verbatim, including potentially three layers of inner exceptions, may include API response body / partial keys." | Medium | The webapp's problem+json `detail` field is generated by ASP.NET Core's `ProblemDetailsFactory` and excludes stack traces / inner exceptions by default. SDK consumers should rely on `detail` + status code for UX, not on free-form `e.Message`. |

## Suggested wrapper companion changes

When migrating to consume the updated webapp:

1. **Send `Idempotency-Key` on every POST through the SDK** so double-clicks/retries don't allocate duplicate licenses. Suggested key: `ReceiptCode + "|" + Computer.MacAddress + "|" + ProductId`.
2. **Read `X-Correlation-Id`** from every response (via SDK) and stash it on `LicenseViewModel` / `ErrorViewModel` so the UI can show it next to "Try again".
3. **Read `Retry-After`** on 429 and surface "retrying in N seconds" countdown in the spinner.
4. **Parse problem+json** error bodies through the SDK; map `status` → exception type and use `detail` as the user-facing message.
