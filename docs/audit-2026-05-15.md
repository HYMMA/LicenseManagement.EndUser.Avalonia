## Executive Summary

This is a small, focused Avalonia 11 UI library (~14 source files, ~1,100 LOC + ~480 lines of AXAML) that wraps the `LicenseManagement.EndUser` SDK to provide ready-made license-management Windows for end-user desktop applications. The package targets `net8.0-windows` but depends on a NuGet package that ships **only `net481` assemblies** (mismatch suppressed via `NoWarn=NU1701`), which is the single biggest architectural risk in the repo.

Findings span seven lenses:

| Severity | Count | Notable themes |
|---|---|---|
| Critical | 4 | Sync-over-network on UI thread; async-void exception swallowing; NU1701 framework mismatch; `LicenseControl` swallowing all exceptions |
| High | 11 | Public setters on credentials; `ICommand` instances re-allocated on every getter; missing async API usage; `CanExecuteChanged` never raised; broken setter pattern in `LicenseControl.License`; sample app uses `Subscribe(...)` without disposing |
| Medium | 14 | `ConvertBack` throwing `NotImplementedException` everywhere; magic colour hex literals scattered through XAML and C#; mock data in sample app pretends to be real; converters declared at both `Theme` and `UserControl` scope (duplicated keys); `BoolToVisibilityConverter` returns the boolean itself; no logging; no observability on SDK failures; non-cross-platform target framework given README claim of cross-platform support |
| Low / Info | 10 | XML doc comments missing on public surface; `ProductViewModel`-equality not implemented; default `WindowStartupLocation="CenterOwner"` on `RegisterLicenseView`/`UnregisterView` even though they are created without an owner; emoji literals as iconography; `SetProperty` reference comparison on value types; `Created`/`Updated` exposed on `BaseViewModel` (feature envy) |

**Top three concerns:**

1. **Blocking the UI thread on network I/O.** Every `HandleLicense()` call site is synchronous and runs on the dispatcher thread. The SDK README documents an async `HandleLicenseAsync()` overload. Under realistic conditions (slow network, captive portal, DNS timeout) the entire app will freeze for up to 100 seconds.
2. **Async-void event handlers that lose exceptions.** `ShowRegisterWindow` and `ShowUnregisterWindow` in `LicenseViewModel` are `async void`. Any exception there crashes the process instead of surfacing in `ShowErrorView`.
3. **Framework mismatch (net481 тЖТ net8.0).** A net481-only dependency loaded into a net8.0 host works *most of the time* but breaks on type-forwards, WMI initialisation, `System.Web` shims, and `Registry` access on non-Windows platforms тАФ the very platforms the README claims support for.

## Lens 1 тАФ Clean Code

### Critical

- **Severity:** Critical
  **File:** `Views/LicenseControl.axaml.cs:59-67`
  **Problem:** The `Initialize` method's `catch` block has no `when`, no exception parameter, and silently swallows every failure mode тАФ including `ApiException`, `CouldNotReadLicenseFromDiskException`, `LicenseExpiredException`, and even `OutOfMemoryException`. The user is then shown a `LicenseViewModel` built from whatever partial state `HandlingContext` happened to hold.
  **Fix:** Catch specific SDK exceptions (`ApiException`, `ComputerOfflineException`, `LicenseExpiredException`, `ReceiptExpiredException`, `CouldNotReadLicenseFromDiskException`), log them, and propagate via `License = errorVm` or an event. Never `catch { }` without acting.
  **Rationale:** A library control that silently eats every exception cannot be diagnosed in the field. Customers will report "the licence panel is wrong" and there will be no log trail.

- **Severity:** Critical
  **File:** `ViewModels/LicenseViewModel.cs:241, 270`
  **Problem:** `ShowRegisterWindow` and `ShowUnregisterWindow` are declared `private async void`. Any exception escaping them (e.g. from `ShowDialog`, `CheckLicenseFile`) is raised on the synchronisation context as an unobserved exception and crashes the process on .NET 8.
  **Fix:** Change to `private async Task` and wrap the body in `try/catch (Exception ex) { ShowErrorView(ex, control); }`. Bind via an async-aware command (CommunityToolkit.Mvvm `[RelayCommand]` is already in the project dependencies).
  **Rationale:** `async void` is reserved for top-level event handlers; using it for command handlers is a documented anti-pattern (Stephen Cleary).


### High

- **Severity:** High
  **File:** `ViewModels/LicenseViewModel.cs:168-170`
  **Problem:** `ShowRegisterViewCommand`, `ShowUnregisterViewCommand`, and `RenewLicenseFileCommand` allocate a **new `RelayCommand` on every getter access**. The same problem exists in `RegisterLicenseViewModel.cs:39-40` and `UnregisterViewModel.cs:30-31`. Each XAML binding evaluation produces a fresh instance, which (a) defeats `CanExecuteChanged` (b) wastes allocations on every property-changed notification (c) prevents commands being toggled via `RaiseCanExecuteChanged()`.
  **Fix:** Back each command with a private readonly field initialised in the constructor or with a property-initialiser (`public ICommand X { get; } = new RelayCommand(...)`).
  **Rationale:** With compiled bindings, the binding evaluator may keep references to the *first* instance it saw, which means later toggles to `ApiKey` or `IsProcessing` won't refresh the command's `CanExecute`. This is invisible at design time and breaks at runtime under specific timing.

- **Severity:** High
  **File:** `Views/LicenseControl.axaml.cs:18-26`
  **Problem:** The `License` property setter writes to `LicenseProperty` **and** assigns to `DataContext`. But the `StyledProperty<>` callback path (binding from outside via `<LicenseControl License="{Binding ...}"/>`) goes through `SetValue` directly without re-entering the CLR setter, so DataContext is **not** updated when set via binding тАФ only when set imperatively. Worse, the `OnAttachedToVisualTree` fix-up only runs once and reads `License` before the binding may have propagated.
  **Fix:** Use the StyledProperty changed callback: `LicenseProperty.Changed.AddClassHandler<LicenseControl>((c, e) => c.DataContext = e.NewValue.Value);`. Or set `DataContext="{Binding $self.License}"` in XAML.
  **Rationale:** Imperative `Initialize(...)` callers work; everyone using XAML bindings sees an empty card.

- **Severity:** High
  **File:** `ViewModels/RegisterLicenseViewModel.cs:15-19`, `UnregisterViewModel.cs:13-16`, `LicenseViewModel.cs:158, 166`
  **Problem:** `ApiKey`, `PublicKey`, `VendorId`, `ProductId` are exposed as `public` settable strings on view-models. They are not raised via `INotifyPropertyChanged` so external mutations are silently invisible to the view, and they are reachable from any code holding a reference to the VM. The same fields are also passed across constructors as plain strings, so the secret travels through at least three objects per registration.
  **Fix:** Group into an immutable `LicenseCredentials` record (a single value-object) with non-nullable required properties, constructed once at the boundary. Make them `init`-only or behind an internal factory.
  **Rationale:** Primitive obsession + leakable secrets. Today `ApiKey` is stored on three separate VM instances and copied between them via property assignment тАФ any debugger dump will show it three times.

- **Severity:** High
  **File:** `samples/AvaloniaSampleApp/MainWindow.axaml.cs:39, 42-49`
  **Problem:** `this.GetObservable(BoundsProperty).Subscribe(new BoundsObserver(this))` тАФ the `IDisposable` returned by `Subscribe` is discarded, so the observer is held by the property system for the lifetime of the window. The `BoundsObserver` strongly references the window. Combined with `MainWindow` being potentially recreated (rare here but realistic for `LicenseWindow` in real apps), this is a textbook leak pattern.
  **Fix:** Store the subscription in a field and dispose it in `OnClosed`. Or simply override `OnSizeChanged`/use the framework-level event.
  **Rationale:** Avalonia subscriptions returned from `GetObservable` must be disposed; otherwise they live forever.

- **Severity:** High
  **File:** `Commands/RelayCommand.cs:29-31`
  **Problem:** `RaiseCanExecuteChanged()` exists but **is never called from anywhere**. Toggling `IsProcessing` in `RegisterLicenseViewModel` does not re-evaluate `CanRegisterNewLicense`. The "Register" button stays clickable while processing.
  **Fix:** Either (a) call `((RelayCommand)RegisterCommand).RaiseCanExecuteChanged()` after every `IsProcessing` change, or (b) migrate to `CommunityToolkit.Mvvm`'s `RelayCommand` which already wires this. The package is already referenced.
  **Rationale:** The `IsEnabled="{Binding !IsProcessing}"` in XAML masks this for the *button* state, but the command itself still executes if invoked programmatically or via keyboard accelerator.

- **Severity:** High
  **File:** `ViewModels/RegisterLicenseViewModel.cs:59`, `UnregisterViewModel.cs:44`, `LicenseViewModel.cs:176, 201`
  **Problem:** `new PublisherPreferences(VendorId!, ProductId!, ApiKey!)` uses the **null-forgiving operator** liberally on user-facing input. If credentials are not configured (a common dev scenario), the SDK will throw an obscure NRE deep inside its constructor instead of a clear "credentials not set" error.
  **Fix:** Validate at the boundary: `if (string.IsNullOrWhiteSpace(VendorId)) throw new InvalidOperationException("VendorId must be set before calling Register.");`. Or, again, an immutable credentials record removes the nullability.
  **Rationale:** `!` is a promise to the compiler. Promises broken at runtime produce the worst-quality error messages.

- **Severity:** High
  **File:** `ViewModels/BaseViewModel.cs:42-58`
  **Problem:** `ShowErrorView(Exception, Control?)` is on the base view-model. View-models should not know about views. This also creates a cyclic dependency: `BaseViewModel` тЖТ `Views.ErrorView` тЖТ `ErrorViewModel` тЖТ `BaseViewModel`.
  **Fix:** Inject an `IErrorPresenter` (or `IDialogService`) into VM constructors, register the implementation in the host's DI/composition root.
  **Rationale:** Untestable: you cannot unit-test `RegisterLicenseViewModel` without spinning up Avalonia.

- **Severity:** High
  **File:** `samples/AvaloniaSampleApp/MainWindow.axaml.cs:15-19`
  **Problem:** Placeholder values like `"your-api-key-here"` and `"<RSAKeyValue><Modulus>your-public-key-modulus</Modulus>..."` are `const string`s in the **sample app**. They look like real credentials at a glance, and any user copy-pasting the sample will run a "register" against a clearly-invalid public key тАФ producing a confusing SDK error instead of a clean "configure me first" message.
  **Fix:** Either move these into `appsettings.json`, or wrap the `RegisterLicense` flow with an early "credentials look like placeholders" check that shows a friendlier error.
  **Rationale:** Sample apps are documentation. They should not appear functional with broken credentials.

- **Severity:** High
  **File:** `ViewModels/LicenseViewModel.cs:30-66`
  **Problem:** Static factory `FromContext` does five distinct things (build VM, conditionally hydrate Receipt, conditionally hydrate Product, conditionally hydrate Computer, assign Products). Method length is 36 lines and its name reads as a builder. Method does not handle `context.LicenseModel` being null (it directly accesses `context.LicenseModel.TrialEndDate`).
  **Fix:** Split into private `MapReceipt(LicenseModel)`, `MapProduct(LicenseModel)`, `MapComputer(LicenseModel)` helpers, or use a dedicated mapper class.
  **Rationale:** This is the central hot-path during licence refresh; it must be readable.


### Medium

- **Severity:** Medium
  **File:** `Converters/VisibilityConverters.cs:8-21`
  **Problem:** `BoolToVisibilityConverter` returns the boolean itself тАФ i.e. it is an identity converter that does no real work. `InverseBoolToVisibilityConverter` returns the negation. Avalonia binds `IsVisible` directly to bool already; these converters add no value. Worse, the class is also misnamed тАФ Avalonia does not have a `Visibility` enum, so the name is borrowed from WPF.
  **Fix:** Delete both classes and the resource keys in `Themes/LicenseTheme.axaml` (lines 29-30). Bind directly: `IsVisible="{Binding IsProcessing}"` or `IsVisible="{Binding !IsProcessing}"`.

- **Severity:** Medium
  **File:** `Converters/*.cs` (every file)
  **Problem:** Every `IValueConverter.ConvertBack` throws `NotImplementedException`. While one-way bindings do not trigger `ConvertBack`, **any binding to a TextBox or any two-way default** will throw. The compiler will not catch this тАФ it is a runtime XAML failure.
  **Fix:** Return `BindingOperations.DoNothing` (or `AvaloniaProperty.UnsetValue` for Avalonia) тАФ the documented "I do not convert back" sentinel. Only throw if you actively want to detect misuse.

- **Severity:** Medium
  **File:** `Themes/LicenseTheme.axaml:16-30`, `Views/LicenseControl.axaml:16-27`
  **Problem:** The same converter keys (`LicenseStatusToTextConverter`, etc.) are declared **both** in the theme's `Styles.Resources` *and* in the user control's `UserControl.Resources`. The latter wins via inner scope, the former is dead, and any maintainer who edits one wonders why their change is not taking effect.
  **Fix:** Pick one location (the theme is correct since the keys are reusable) and delete the duplicates from `LicenseControl.axaml`.

- **Severity:** Medium
  **File:** `ViewModels/RegisterLicenseViewModel.cs:97-99`
  **Problem:** `ChangeDefaultTrial(PublisherPreferences)` and `ComputerUnregistered(ComputerModel)` are empty method bodies passed as SDK callbacks. The user-facing behaviour for "trial ended" and "computer unregistered" is silently nothing.
  **Fix:** Either decide explicitly (show a message, log) or document why they are intentionally empty with a comment + `[SuppressMessage]`.

- **Severity:** Medium
  **File:** `ViewModels/ErrorViewModel.cs:14-23`
  **Problem:** `CloseCommand = new RelayCommand(_ => { }, _ => true)` тАФ the command does nothing. The actual close is wired via code-behind (`ErrorView.axaml.cs` `CloseButton_Click`). So the command is dead code, and the button in XAML is mis-wired to a `Click` event instead of `Command`.
  **Fix:** Either remove the command (and keep `Click`) or remove `Click` and have the command close the parent window via a passed parameter, like the other windows do.

- **Severity:** Medium
  **File:** `ViewModels/BaseViewModel.cs:12-25`
  **Problem:** `Created` and `Updated` `DateTime` properties live on the base class but only `LicenseViewModel` and `ProductViewModel` use them. `ErrorViewModel`, `RegisterLicenseViewModel`, and `UnregisterViewModel` inherit them for free, polluting the public surface and binding scope.
  **Fix:** Move these to a derived `TimestampedViewModel` or just to `LicenseViewModel`/`ProductViewModel` directly.

- **Severity:** Medium
  **File:** `Converters/LicenseStatusConverters.cs:50-55, 75-80`, `Views/LicenseControl.axaml:34`, etc.
  **Problem:** Colour hex literals (`#FFF0F0`, `#F0FFF4`, etc.) are duplicated between C# converters and AXAML markup. The theme already defines `PrimaryBrush`/`DangerBrush`/etc. as resources тАФ but the converters bypass them by parsing strings.
  **Fix:** Have the converters resolve resources via `Application.Current.Resources["DangerBrush"]`, or, better, make the converter return a *resource key* and have the binding resolve. Centralises theming.

- **Severity:** Medium
  **File:** `Views/LicenseControl.axaml:38, 102`, `Views/RegisterLicenseView.axaml:26`, `Views/UnregisterView.axaml:26`
  **Problem:** Emoji glyphs (`ЁЯУД`, `ЁЯФС`, `тЪа`) are used as iconography. Font rendering of emoji varies wildly across platforms (Windows segoe-ui-emoji vs macOS Apple Color Emoji vs Linux varied). On systems without a colour-emoji font installed, they render as boxes. The README promises cross-platform support.
  **Fix:** Replace with `PathIcon`/`Geometry` using the Fluent System Icons set (or any SVG), or with `MaterialDesign` font glyphs. Vector icons render identically on every platform.

- **Severity:** Medium
  **File:** `ViewModels/LicenseViewModel.cs:122-132`
  **Problem:** `Product` setter has custom comparison logic (`if (_product == null || _product.Id != value?.Id)`). `SetProperty` already does an `Equals` check. This duplicates work and reads as "we are routing around the default for an unclear reason." If the intent is to compare by ID only, `ProductViewModel` should implement `IEquatable<ProductViewModel>`.
  **Fix:** Implement `IEquatable<ProductViewModel>` on `ProductViewModel` comparing on `Id`, then use the default `SetProperty`.

- **Severity:** Medium
  **File:** `Views/LicenseControl.axaml.cs:38-68`
  **Problem:** `Initialize` accepts `products` as a default-null parameter, then both branches do `products ?? new ObservableCollection<ProductViewModel>()`. Two duplications.
  **Fix:** Coerce once at the top: `products ??= new ObservableCollection<ProductViewModel>();`.

## Lens 2 тАФ Design Patterns

### High value

1. **Strategy / lookup table on LicenseStatus тЖТ UI behaviour** тАФ Currently four converters (`LicenseStatusToText`, `тАжBackground`, `тАжBorder`, `тАжIcon`) and four `Visibility` converters all switch over the same enum. Each is a parallel `switch` expression of six cases. Adding a new status requires editing eight files.
   - Refactor: Single `LicenseStatusPresenter` class with one method per facet (`GetText`, `GetBackground`, `GetIcon`, `IsRegisterVisible`, etc.), backed by a `static readonly Dictionary<LicenseStatusTitles, LicenseStatusPresentation>` table. Converters become 6-line delegations.
   - Effort: M. Files affected: all 4 converter files + `LicenseTheme.axaml`.

2. **Command pattern (CommunityToolkit.Mvvm)** тАФ `CommunityToolkit.Mvvm 8.2.2` is already in `PackageReference` but unused. Migrating `RelayCommand` and `ObservableObject` from hand-rolled to the toolkit eliminates `BaseViewModel`, the broken `RaiseCanExecuteChanged` plumbing, and the entire `Commands/RelayCommand.cs` file тАФ and gives `[RelayCommand]` source-generator-backed async commands that capture exceptions properly.
   - Effort: M. Files affected: `BaseViewModel.cs`, `Commands/RelayCommand.cs`, every `*ViewModel.cs`.

3. **Mediator / Service layer between VM and SDK** тАФ The view-models directly instantiate `PublisherPreferences`, `LicHandlingContext`, and `LicenseHandlingLaunch`/`Install`/`Uninstall`. This couples the VMs to SDK constructors, makes them un-mockable, and means any SDK breaking change requires touching every VM.
   - Refactor: Introduce `ILicenseService` with `Task<LicenseModel> RegisterAsync(...)`, `Task<LicenseModel> UnregisterAsync(...)`, `Task<LicenseModel> RefreshAsync(...)`. VMs call the service; the service constructs the SDK handlers.
   - Effort: M-L. Files affected: new `Services/ILicenseService.cs`, new `Services/LicenseService.cs`, all 3 action VMs.

### Medium value

4. **Result<T> instead of throw-catch-display** тАФ Every register/unregister/refresh wraps SDK calls in `try/catch (Exception)` and pushes the message into `ErrorMessage` (UI) or `ShowErrorView`. A `Result<LicenseModel, LicenseError>` type would let the VM branch on error kind without `catch`-driven control flow.
   - Effort: M.

5. **Builder for `LicenseViewModel.FromContext`** тАФ Already a large factory; a fluent builder taking the `LicHandlingContext` and producing a hydrated VM step-by-step would improve readability.
   - Effort: S.

### What NOT to add

- A factory for `LicenseHandlingLaunch`/`Install`/`Uninstall` тАФ only three classes, no variation; over-engineering.
- A visitor over `LicenseStatusTitles` тАФ the Strategy table is sufficient.

### Suggested refactoring order

1. Adopt `CommunityToolkit.Mvvm` (deletes code and removes bugs simultaneously).
2. Extract `ILicenseService` (testability unlock).
3. Collapse the converter zoo into one presenter.

## Lens 3 тАФ Security Review

### Critical

- **Severity:** Critical
  **File:** `samples/AvaloniaSampleApp/MainWindow.axaml.cs:17-18`
  **Problem:** `ApiKey` is declared as `private const string ApiKey = "your-api-key-here";`. While this is the *sample*, the pattern (compile-time-baked credential) is what consumers will copy into their production app. The README's Quick Start does the same thing.
  **Fix:** Update both the sample and the README to read credentials from `appsettings.json`, environment variables, or platform-secure storage (Windows Credential Manager / DPAPI / macOS Keychain). Add a `// DO NOT hardcode in production` banner.
  **Rationale:** Hardcoded API keys end up in source control. Even a "client API key" grants enough access to enumerate licences for a publisher.

### High

- **Severity:** High
  **File:** `ViewModels/LicenseViewModel.cs:158`
  **Problem:** `public string? ApiKey { get; set; }` тАФ the API key is a public property with a public setter and is **not marked as a secret**. Avalonia's data-context inspector (and `Avalonia.Diagnostics`, which is enabled in the sample app `Debug` config at line 16 of the sample csproj) will display this in plaintext on any developer's machine.
  **Fix:** Mark with `[Browsable(false)]` and `[DebuggerBrowsable(DebuggerBrowsableState.Never)]`. Better: make the setter `internal`, or hold it via a `Func<string>` provider so the raw value never sits on the VM.
  **Rationale:** Avalonia's dev tools dump all DataContext properties on right-click. Customer support sessions and screen recordings will leak keys.

- **Severity:** High
  **File:** `ViewModels/RegisterLicenseViewModel.cs:21-25`
  **Problem:** `ReceiptCode` (which is the customer's product key) is bound to a regular `TextBox` (`Views/RegisterLicenseView.axaml:39-44`) with no masking, no clipboard-clear, and no paste-from-clipboard handling. Anyone shoulder-surfing sees the full key.
  **Fix:** Use the `PasswordChar`/`RevealPassword` pattern тАФ Avalonia 11 supports `<TextBox PasswordChar="*"/>` with a reveal toggle.

- **Severity:** High
  **File:** `Views/LicenseControl.axaml.cs:46-50`
  **Problem:** `PublicKey` (RSA `<RSAKeyValue>` blob) is constructed into `PublisherPreferences` at every `Initialize`. While the public key itself is not secret, the **pattern** of accepting it from a caller-provided string allows an attacker who can MITM the application to swap the public key and forge licences. There is no pinning or fingerprint validation.
  **Fix:** Embed the public key as an embedded resource in the consuming application's assembly, not a parameter. Or at minimum, hash-validate the supplied key against a known fingerprint.

### Medium

- **File:** `ViewModels/ErrorViewModel.cs:20-22` тАФ Exception messages are shown verbatim, including potentially three layers of inner exceptions. SDK exceptions from `ApiException` may include the API response body, which can include request IDs / tokens / partial keys. The error view renders this directly in the UI for the end user.
  **Fix:** Sanitise the message via a whitelist of "safe" exception types; show a generic message otherwise; log the full detail.

- **File:** `Views/RegisterLicenseView.axaml:39-44` тАФ The `ReceiptCode` TextBox uses `FontFamily="Consolas, monospace"` and `Watermark="XXXX-XXXX-XXXX-XXXX"`. It does not clear the clipboard on focus loss, does not have `IsUndoEnabled="False"` (so the key sits in the undo stack), and is not marked `ContextMenu="{x:Null}"` (right-click "Copy" works after entry).
  **Fix:** Disable undo, disable context menu, optionally clear clipboard after successful registration if the key came from clipboard.

- **File:** `ViewModels/BaseViewModel.cs:42-58` тАФ `ShowErrorView(ex, owner)` calls `errorView.Show()` (non-modal) when no owner is found. On Windows, this means the error window can be dismissed by the OS (lost focus) before the user sees it, and the underlying error never reaches a log.
  **Fix:** Always log via `ILogger` (or `Trace.WriteLine`) before showing UI.

### Info

- **File:** `LicenseManagement.EndUser.Avalonia.csproj:5` тАФ `<Nullable>enable</Nullable>` is on. Good. But the codebase uses `!` in 11 places to bypass it, eroding the value.
- **File:** project-wide тАФ No logging dependency. `Microsoft.Extensions.Logging.Abstractions` would let the library participate in the host's log stack without imposing a logger.

## Lens 4 тАФ External Integrations Review

The library wraps **one** external SDK (`LicenseManagement.EndUser`) which talks to `license-management.com`. There are no direct `HttpClient` calls, Stripe, AWS, etc. тАФ but the SDK is itself an external-integration boundary.

### Critical

- **Severity:** Critical
  **File:** `ViewModels/RegisterLicenseViewModel.cs:79`, `UnregisterViewModel.cs:56`, `LicenseViewModel.cs:187, 212`, `Views/LicenseControl.axaml.cs:61`
  **Problem:** All five call sites call **`handler.HandleLicense()`** (the synchronous overload), even though the SDK README documents an async overload `HandleLicenseAsync()`. The synchronous call blocks the calling thread on network I/O. In every one of these call sites, the calling thread is the **UI dispatcher thread**. The default `HttpClient` timeout is 100 seconds тАФ meaning the app can hang for 100 s on a slow connection or captive-portal scenario before the user gets any feedback.
  **Fix:** Replace every call site with `await handler.HandleLicenseAsync()` (the SDK README shows this exact API). Promote the surrounding methods to `async Task`. Wire commands via CommunityToolkit `[RelayCommand]` which already supports async.
  **Rationale:** Frozen UIs are the single most common complaint reported against desktop license-management dialogs. The SDK already provides the fix; the wrapper just does not use it.

### High

- **Severity:** High
  **File:** Every catch block on SDK calls (`RegisterLicenseViewModel.cs:81-86`, `UnregisterViewModel.cs:58-63`, `LicenseViewModel.cs:189-194, 213-217`, `LicenseControl.axaml.cs:63-67`)
  **Problem:** Every catch is `catch (Exception e)` or `catch { }`. The SDK README documents a specific exception taxonomy: `ComputerOfflineException`, `CouldNotReadLicenseFromDiskException`, `LicenseExpiredException`, `ReceiptExpiredException`, `ApiException` (with `StatusCode`). None of these are branched on тАФ the user sees the same "show error dialog" path regardless of root cause.
  **Fix:** Branch on the documented exceptions and provide differentiated UX:
    - `ComputerOfflineException` тЖТ retry button, "check your internet"
    - `LicenseExpiredException` тЖТ "renew" CTA
    - `ReceiptExpiredException` тЖТ "your subscription has ended" + link to publisher
    - `ApiException` with 429 тЖТ "rate limited, retrying in N seconds"
    - `ApiException` with 5xx тЖТ "service temporarily unavailable, try later"
    - `CouldNotReadLicenseFromDiskException` тЖТ "reinstall" hint
  **Rationale:** The error catalogue is the API; ignoring it defeats the whole point of a typed exception hierarchy.

- **Severity:** High
  **File:** `LicenseManagement.EndUser.Avalonia.csproj:24, 42-47`
  **Problem:** The project targets `net8.0-windows`. The `LicenseManagement.EndUser` 2.0.0 NuGet ships **only** `lib/net481/LicenseManagement.EndUser.dll` (.NET Framework 4.8.1). NuGet emits warning `NU1701` ("Package may not be fully compatible") тАФ the project suppresses it via `<NoWarn>$(NoWarn);NU1701</NoWarn>`. This works by accident under net8.0 + Windows because most of the surface area uses BCL APIs that have compatible signatures, but it will break the moment the SDK uses something like `System.Web` or `System.Drawing.Common` that is not compat-shimmed.
  **Fix:** Either (a) ship a multi-targeted SDK (request from upstream), or (b) target `net48`/`net481` from this wrapper too, or (c) lift the NU1701 suppression to confirm the runtime contract still holds with every SDK update.
  **Rationale:** The README claims "Cross-platform: Works on Windows, macOS, and Linux" but the dependency uses Windows Registry + WMI per its own README, so the cross-platform claim is **objectively false** today. Either fix the dependency story or update the README.

### Medium

- **File:** `ViewModels/RegisterLicenseViewModel.cs:47-86` тАФ No idempotency key, no retry, no timeout policy. If the user clicks "Register" twice (which is possible because `CanExecute` does not refresh тАФ see clean-code finding above), the SDK is asked to register twice, potentially burning a license seat.
  **Fix:** Disable the button via `IsEnabled` (already done in XAML) **and** raise `CanExecuteChanged` (not done) **and** ideally derive an idempotency token from `(ReceiptCode + Computer.MacAddress)`.

- **File:** `ViewModels/LicenseViewModel.cs:172-195` тАФ `CheckLicenseFile` is called from the UI on `OnRegistrationComplete` and after every dialog dismissal. There is no rate-limiting; if the user opens and closes the register window 10 times in a row, that is 10 network round-trips.
  **Fix:** Throttle via `DispatcherTimer` or cache the last result for ~30 seconds.

- **File:** project-wide тАФ no observability. There is no `ILogger` injection, no `EventSource`, no telemetry. When license-registration fails in the field, the *only* signal is the user's verbatim exception text relayed by support.
  **Fix:** Add `Microsoft.Extensions.Logging.Abstractions` (header-only); inject `ILogger<LicenseViewModel>`; log every SDK entry/exit with elapsed time, status code, and user-facing outcome.

- **File:** every SDK call site тАФ there is no operation/correlation ID propagated. If a customer reports "I tried to register at 14:32 UTC", support has no way to find the corresponding API request.
  **Fix:** Generate a `Guid` per logical operation, pass it via `PublisherPreferences` metadata (the SDK supports `BeforeLicensePost` metadata events per its README), surface it in the error UI for support copy-paste.

### Low

- **File:** `ViewModels/LicenseViewModel.cs:185-194` тАФ `catch (Exception)` discards the exception object, then accesses `context.Exception` which is the SDK's own captured exception. This is a fragile contract тАФ if the SDK ever stops populating `context.Exception`, the error view shows nothing.
  **Fix:** Use the caught `Exception` instead of routing through context state.

## Lens 5 тАФ Async / Threading

### Critical (already listed above, repeated for visibility)

- **`LicenseHandlingLaunch.HandleLicense()` on UI thread** тАФ see Lens 4 #1.
- **`async void ShowRegisterWindow` / `ShowUnregisterWindow`** тАФ see Lens 1 Critical #2.

### High

- **Severity:** High
  **File:** `ViewModels/RegisterLicenseViewModel.cs:55, 73, 82-86`
  **Problem:** `IsProcessing = true` then **synchronous** `handler.HandleLicense()` then various `OnRegistrationComplete?.Invoke()`. Because the call is sync on the UI thread, `IsProcessing = true` never paints тАФ the UI is frozen for the duration. The loading indicator (`<Border>` with the spinning border in `RegisterLicenseView.axaml:63-68`) is never seen.
  **Fix:** Combine with the async migration. Once the call is `await`-ed, the dispatcher resumes between assignment and the call, the spinner renders, and the UX matches the design.

- **Severity:** High
  **File:** `Views/LicenseControl.axaml.cs:53-57`
  **Problem:** The `OnLicenseHandledSuccessfully` callback is invoked by the SDK on **its** thread (probably a thread-pool worker, since the SDK fetches asynchronously). The callback then does `License = LicenseViewModel.FromContext(...)`. Setting `License` (which is a `StyledProperty`) **on a non-UI thread** will throw `InvalidOperationException: Cross-thread access` in Avalonia.
  **Fix:** Marshal to the dispatcher: `await Dispatcher.UIThread.InvokeAsync(() => { License = ... });`. Same applies to `UpdateFromLicenseModel`.

- **Severity:** High
  **File:** `ViewModels/RegisterLicenseViewModel.cs:71-77`
  **Problem:** The `OnLicenseHandledSuccessfully` callback mutates VM state (`IsProcessing = false`) and calls `window.Close()`. If the SDK invokes this from a background thread (likely), both are cross-thread violations: `IsProcessing` raises `PropertyChanged` synchronously, and any binding evaluation happens on the originating thread; `window.Close()` must be on the UI thread.
  **Fix:** Wrap the callback body in `Dispatcher.UIThread.Post(() => { ... })`.

### Medium

- **File:** `BaseViewModel.cs:27-29` тАФ `OnPropertyChanged` fires on whatever thread the setter is called from. With async migrations, this becomes a correctness footgun. Standard pattern is to marshal in the base.
  **Fix:** In `OnPropertyChanged`, dispatch via `Dispatcher.UIThread.Post` if not on the UI thread.

- **File:** `ViewModels/LicenseViewModel.cs:241, 270` (already noted as Critical for async-void). Also note: after `await view.ShowDialog(parentWindow)`, `CheckLicenseFile(control)` is invoked. If `ShowDialog` returned because the window was closed by another path (e.g. parent closed), `control` may be detached from the visual tree and `TopLevel.GetTopLevel(control)` returns null тАФ silently degrading the refresh path.
  **Fix:** Re-check `control.GetVisualRoot() != null` before refresh.

## Lens 6 тАФ Avalonia / XAML Best Practices

### High

- **Severity:** High
  **File:** `Views/LicenseControl.axaml:9`, all `*View.axaml`
  **Problem:** `x:DataType` is set (good тАФ compiled bindings enabled). But the `UserControl.Resources` block in `LicenseControl.axaml` re-declares converters that already exist in the merged theme. This double-registration is silently allowed but means inserting a *new* converter in the theme has no effect if the same key exists on the user control.
  **Fix:** Remove `UserControl.Resources` converter declarations тАФ they are already in `LicenseTheme.axaml`.

- **Severity:** High
  **File:** `Views/LicenseWindow.axaml:13`
  **Problem:** `SizeToContent="Height"` on a `Window` with a single child whose layout depends on `MinHeight` / runtime data. On first show with empty data, the window can collapse to under 100 px. Customer reports of "the dialog opens tiny" usually trace to this.
  **Fix:** Add `MinHeight="400"` (or appropriate). Same for `ErrorView`, `RegisterLicenseView`, `UnregisterView`.

- **Severity:** High
  **File:** `Themes/LicenseTheme.axaml:74-76, 89-91, 103-105`
  **Problem:** Selectors like `Button.primary:pointerover /template/ ContentPresenter` target the **template internals** of `Button`. This works on Avalonia 11.x's *current* default template, but is fragile across versions тАФ Avalonia 12 already changes some template part names. The `Setter` directly on `Button.primary` is the safer pattern.
  **Fix:** Use `Button.primary:pointerover { Setter Property="Background" ... }` without the `/template/` descent. Avalonia inherits Background through the default template.

### Medium

- **File:** `Views/LicenseControl.axaml:138, 146, 154` тАФ `CommandParameter="{Binding #ThisControl}"` (element reference). This works, but it couples the command handler to the *view*, which is what `IDialogService`/`IErrorPresenter` was meant to avoid.
  **Fix:** Long-term, move dialog presentation to an injected service. Short-term, this is fine.

- **File:** `Views/LicenseControl.axaml:46-49` тАФ `Background="{Binding Status, Converter={StaticResource LicenseStatusToBackgroundConverter}}"` on a `Border` whose parent `WrapPanel` may wrap unexpectedly under width constraints. The badge can wrap to a second line with the title.
  **Fix:** Constrain `WrapPanel` to single-line via a `DockPanel` or fixed-row `Grid` for the header.

- **File:** `Views/LicenseControl.axaml:113-117` тАФ `ComboBox.ItemTemplate` with a `DataTemplate` lacking `IsItemItsOwnContainer` / explicit `ItemContainerTheme`. Default container styling applies, which is fine, but the `TextBlock` has `Padding="4"` only тАФ the dropdown items will look cramped on touch devices.
  **Fix:** Increase padding or set `ItemContainerTheme`.

- **File:** `Views/RegisterLicenseView.axaml:63-68` тАФ The "spinner" is a static `<Border>` with a `RotateTransform` that has no animation. It does not actually spin.
  **Fix:** Add a `Storyboard`/`Animations` block with an `Animation` rotating the transform from 0 to 360 in 1s on loop.

### Low

- **File:** `Themes/LicenseTheme.axaml:1-3` тАФ namespace declaration includes `xmlns:converters` but the file mixes resources and styles in the same `<Styles>` root. Avalonia accepts this but it is idiomatic to split into separate `.axaml` files (`Converters.axaml`, `Buttons.axaml`, `Cards.axaml`) for maintainability.
- **File:** every `*View.axaml` тАФ `mc:Ignorable="d"` and `d:DesignWidth/Height` are present. Good.
- **File:** `Views/ErrorView.axaml:13` тАФ `WindowStartupLocation="CenterOwner"` but the view may be shown without owner (`BaseViewModel.ShowErrorView` line 56 path). Add a fallback to `CenterScreen`.

## Lens 7 тАФ MVVM Hygiene

### High

- **File:** `Views/ErrorView.axaml.cs:13-16` тАФ `CloseButton_Click` in code-behind. The `ErrorViewModel` already has a `CloseCommand` (dead code at the moment per Lens 1). Code-behind handlers couple view to specific button instance and bypass the command surface.
  **Fix:** Wire the button via `Command="{Binding CloseCommand}"` with the close logic in the VM (close via passed-in window parameter, as other views do).

- **File:** `samples/AvaloniaSampleApp/MainWindow.axaml.cs:91-107` тАФ `UpdateFeatureButtons` directly toggles `IsEnabled` on named XAML elements. This is the WPF/WinForms idiom; in MVVM you bind `IsEnabled` to a VM property (`HasPaidLicense`, `HasAnyLicense`) and let the binding system do it.
  **Fix:** Move to a `SampleAppViewModel` with `HasPaidLicense`/`HasAnyLicense` properties.

### Medium

- **File:** `ViewModels/LicenseViewModel.cs` тАФ VM both owns state (DTO-ish) and orchestrates side effects (showing dialogs, calling SDK). This is the "Smart VM" anti-pattern. Splitting into `LicenseStatus` (record/DTO) + `LicenseInteractor` (commands) + `LicensePresenter` (formatting) is cleaner.
- **File:** `Views/LicenseControl.axaml.cs:38-68` тАФ `Initialize(...)` is **imperative configuration on a UI control**. Two consequences: (a) the control can be configured only once, and (b) the configuration is not visible in XAML so consumers cannot preview it. The idiomatic Avalonia approach is to expose dependency properties (`VendorId`, `ProductId`, etc.) and react to them.
  **Fix:** Promote each parameter to a `StyledProperty<string?>`; react to changes in a single `OnPropertyChanged` override.

## Cross-cutting Themes

1. **Async story is unrealized.** The SDK exposes async APIs (`HandleLicenseAsync`). The wrapper uses none of them. Every UI operation that touches the network blocks the dispatcher. Fixing this alone resolves three Critical findings and changes the UX from "hangs randomly" to "shows progress, then completes."

2. **`CommunityToolkit.Mvvm` is referenced but not used.** Adopting it deletes `BaseViewModel.cs`, `Commands/RelayCommand.cs`, and roughly 40 lines of property boilerplate per VM. It also fixes the `CanExecuteChanged` and async-command issues for free.

3. **Cross-platform claim vs. Windows-only dependency.** README says "Works on Windows, macOS, and Linux." The csproj targets `net8.0-windows`. The dependency ships only `net481`, uses `System.Management` (WMI) and `Microsoft.Win32` (Registry) per its README. Pick one story: either the SDK adds a `netstandard2.0` build and the wrapper targets `net8.0`, or the README drops the cross-platform claim.

4. **Error catalogue is ignored.** The SDK ships a precise exception hierarchy. The wrapper flattens it to `catch (Exception)`. This is the single most actionable engineering fix on a per-line basis.

5. **Magic constants everywhere.** Colour hex literals appear in 4 converters + the theme + every view. Status text strings ("Trial and active.", etc.) sit in `LicenseStatusToTextConverter` as `const` declarations rather than resource strings тАФ defeating localisation.

6. **Sample app double-codes layouts.** `MainWindow.axaml` declares two parallel layouts (`WideLayout`, `NarrowLayout`) and the code-behind copies state between them. A `Panel.IsVisible` + a single `ResponsivePanel` or `ContentControl` switched by a converter would halve the XAML.

## Prioritised Remediation Roadmap

The roadmap is ordered for maximum impact-per-hour. Each step compounds тАФ done in order, each step is cheaper because the prior one cleaned up.

### Phase 1 тАФ Stop the bleeding (1-2 days)

1. **Migrate all 5 SDK call sites to `HandleLicenseAsync()`** and make their callers `async Task`. (Lens 4 Critical / Lens 5 Critical)
2. **Convert `ShowRegisterWindow` / `ShowUnregisterWindow` from `async void` to `async Task`** and bind via async command. (Lens 1 Critical / Lens 5 Critical)
3. **Wrap every SDK callback that mutates VM state in `Dispatcher.UIThread.Post(...)`** to fix cross-thread access. (Lens 5 High)
4. **Replace `catch (Exception)` with the specific SDK exceptions** (`ApiException`, `ComputerOfflineException`, `LicenseExpiredException`, `ReceiptExpiredException`, `CouldNotReadLicenseFromDiskException`) with differentiated handling. (Lens 4 High)
5. **Stop swallowing errors silently in `LicenseControl.Initialize`** тАФ at minimum, log and re-expose via an event or property. (Lens 1 Critical)

### Phase 2 тАФ Adopt the toolkit you already pay for (1 day)

6. **Migrate `BaseViewModel` and all VMs to `CommunityToolkit.Mvvm`** (`ObservableObject` + `[ObservableProperty]` + `[RelayCommand]`). Deletes `Commands/RelayCommand.cs`, `BaseViewModel.cs`. Fixes `CanExecuteChanged`. (Lens 1 High / Lens 2)
7. **Remove duplicated converter resource declarations** from `LicenseControl.axaml` and rely solely on `LicenseTheme.axaml`. (Lens 1 Medium / Lens 6 High)
8. **Replace `ConvertBack` throws with `BindingOperations.DoNothing`.** (Lens 1 Medium)

### Phase 3 тАФ Architectural cleanup (2-3 days)

9. **Extract `ILicenseService`** with async methods returning `Result<LicenseModel, LicenseError>` (or similar). VMs depend on the interface, not the SDK. Enables unit testing. (Lens 2 High value #3)
10. **Introduce `IErrorPresenter` / `IDialogService`**. Removes `ShowErrorView` from `BaseViewModel`. (Lens 1 High)
11. **Collapse the converter zoo into a `LicenseStatusPresenter`** backed by a static lookup table. (Lens 2 High value #1)
12. **Promote `LicenseControl.Initialize` parameters to StyledProperties** so consumers can configure declaratively in XAML. (Lens 7 Medium)

### Phase 4 тАФ Cross-platform honesty (1-2 days)

13. **Decide the cross-platform story.** Either request the upstream SDK to ship `netstandard2.0` (or `net6.0`) and remove `NU1701` suppression, or update the README to "Windows only (per current SDK)" and rename the package target to make this explicit. (Lens 4 High)
14. **Replace emoji glyphs with vector icons** (Fluent Icons / SVG geometry) so the UI renders identically across platforms. (Lens 1 Medium)
15. **Implement actual responsive layout** in the sample app via a single layout + Grid columns that collapse, rather than two parallel layouts maintained in code-behind. (Lens 7 High / Cross-cutting #6)

### Phase 5 тАФ Observability and security polish (1 day)

16. **Add `Microsoft.Extensions.Logging.Abstractions`** dependency, inject `ILogger<T>` into VMs and the new `LicenseService`. Log every SDK entry/exit with correlation ID, elapsed time, status. (Lens 4 Medium)
17. **Mark `ApiKey` with `[DebuggerBrowsable(Never)]`** and consider a `Func<string>` provider pattern so the raw key is never on the VM. Update README to recommend platform-secure storage. (Lens 3 High)
18. **Add `MinHeight` to all windows** with `SizeToContent="Height"` to prevent tiny-window failure modes. (Lens 6 High)
19. **Update the hardcoded credentials in the sample** to load from `appsettings.json` or environment variables, with a clear "fill these in first" UX. (Lens 3 Critical)

### Phase 6 тАФ Nice-to-haves (as time allows)

20. Animate the loading spinner.
21. Implement `IEquatable<ProductViewModel>` on `Id`.
22. Move `Created`/`Updated` off `BaseViewModel`.
23. XML doc comments on all public types тАФ they ship in the NuGet package.
